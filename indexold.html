Q1

What is the difference between value types and reference types in C#?

C# FundamentalsEasyOriginal detailed notebook
Why asked

This is asked in almost every .NET interview. It tests whether you understand memory layout — a foundation for writing performant, bug-free code.

How it works

Value types (struct, int, bool, double, DateTime) store data directly on the stack. Reference types (class, string, object, array) store a reference on the stack that points to data on the heap. This changes how assignment, comparison, and method passing work.

Practical / Real-world context

If you pass a struct to a method and modify it, the caller sees no change. If you pass a class object, the caller sees the change. This trips up developers daily in large codebases.

Code example
C# / SQL / Angular
// Value type — copied on assignment
int a = 10;
int b = a;
b = 99;
Console.WriteLine(a); // 10 — NOT affected

// Reference type — shared reference
var list1 = new List<int> { 1, 2, 3 };
var list2 = list1;      // both point to same object
list2.Add(4);
Console.WriteLine(list1.Count); // 4 — AFFECTED

// Struct (value type) example
struct Point { public int X, Y; }
Point p1 = new Point { X = 1, Y = 2 };
Point p2 = p1;   // full copy
p2.X = 99;
Console.WriteLine(p1.X); // 1 — unchanged
Visualize it
Final interview answer
Value types store data on the stack directly — assignment creates a full copy. Reference types store a reference on the stack pointing to heap memory — assignment copies the pointer, so both variables share the same object. This means modifying a reference type through one variable affects all variables pointing to it. Structs are value types; classes are reference types. I use structs for small, immutable data like coordinates, and classes for anything with complex state or behaviour.
Q2

Explain the difference between abstract class and interface. When do you use each?

C# FundamentalsMediumOriginal detailed notebook
Why asked

Asked to test your OOP design judgment. Microsoft expects you to know WHEN to use each, not just WHAT they are.

How it works

Abstract class: can have implementation, constructor, fields, protected members. One class can only extend ONE abstract class. Interface: pure contract (in C# 8+ can have default implementations, but conceptually it is a contract). A class can implement MULTIPLE interfaces.

Practical / Real-world context

IDisposable, IEnumerable, IComparable are interfaces — pure contracts any class can adopt. DbContext (EF Core) is an abstract class — it has a LOT of shared logic. Use interface when you want a capability contract. Use abstract class when you share code.

Code example
C# / SQL / Angular
// Interface — "can do" contract
public interface INotifiable {
    void Notify(string message);
    // C# 8+ default implementation allowed
}

// Abstract class — "is a" with shared logic
public abstract class Animal {
    public string Name { get; set; }          // shared state
    public abstract void MakeSound();          // must override
    public void Breathe() => Console.WriteLine("Breathing..."); // shared
}

// A class CAN do both
public class Dog : Animal, INotifiable {
    public override void MakeSound() => Console.WriteLine("Woof");
    public void Notify(string msg) => Console.WriteLine(msg);
}
Visualize it
Final interview answer
An abstract class is a partially implemented base class — it can hold shared logic, fields, and a constructor, but cannot be instantiated directly. An interface is a pure behavioural contract. A class can only inherit one abstract class but implement many interfaces. I choose an abstract class when multiple types share code or state — like a base Entity with audit fields in EF Core. I choose an interface when I want to define a capability that unrelated classes can adopt — like IRepository or ILogger. Since C# 8, interfaces can have default implementations, which blurs the line slightly, but the design intent remains different.
Q3

What is the difference between string and StringBuilder? When should you use each?

C# FundamentalsEasyOriginal detailed notebook
Why asked

Performance question that shows up in code reviews and interviews. Immutability of string is a key C# concept.

How it works

String in C# is immutable — every concatenation creates a NEW string object on the heap. StringBuilder is a mutable buffer — it appends in-place, growing an internal char array. For 2–3 concatenations in a loop of 10,000 iterations, string creates 10,000 garbage objects; StringBuilder creates one.

Practical / Real-world context

Building SQL queries dynamically, generating HTML templates, processing CSV rows — anywhere you concatenate in a loop. String is fine for one-off template literals or interpolation.

Code example
C# / SQL / Angular
// BAD — creates N garbage strings in loop
string result = "";
for (int i = 0; i < 10000; i++)
    result += i.ToString(); // new string every iteration!

// GOOD — one mutable buffer
var sb = new StringBuilder();
for (int i = 0; i < 10000; i++)
    sb.Append(i);
string result2 = sb.ToString(); // one final allocation

// String interpolation is fine for simple cases
string name = "Alice";
string greeting = $"Hello, {name}!"; // compiler optimises this
Visualize it
Final interview answer
String in C# is immutable — every concatenation allocates a new string object on the heap, leaving the old one for the garbage collector. For a handful of static concatenations this is fine and the compiler often optimises it. But inside a loop or when building dynamic content, I always use StringBuilder because it maintains a mutable internal buffer, avoiding the allocation churn. I call .ToString() once at the end. A common rule of thumb: three or fewer concatenations → string; loop or dynamic accumulation → StringBuilder.
Q4

What are delegates, events, and Func/Action? How are they related?

C# FundamentalsMediumOriginal detailed notebook
Why asked

Callbacks and event-driven code power the entire .NET framework — WPF, ASP.NET Core middleware, EF Core interceptors. You must know these cold.

How it works

A delegate is a type-safe function pointer — it holds a reference to a method with a specific signature. An event wraps a delegate with publisher/subscriber semantics (only the owner can invoke it). Func and Action are built-in generic delegates — they save you from declaring custom delegate types.

Practical / Real-world context

Button.Click in WPF, HttpClient message handlers, EF Core SaveChanges events, LINQ Where/Select predicates (Func<T,bool>) — everything that says "call this method later" uses delegates under the hood.

Code example
C# / SQL / Angular
// Custom delegate type
delegate int MathOp(int a, int b);
MathOp add = (a, b) => a + b;
Console.WriteLine(add(3, 4)); // 7

// Func — returns a value:  Func<in, in, out>
Func<int, int, int> multiply = (a, b) => a * b;

// Action — returns void:  Action<in>
Action<string> log = msg => Console.WriteLine(msg);

// Predicate — special Func<T, bool>
Predicate<int> isEven = n => n % 2 == 0;

// Event — wraps MulticastDelegate for pub/sub
public class Button {
    public event EventHandler Clicked;   // only Button can invoke
    public void Click() => Clicked?.Invoke(this, EventArgs.Empty);
}
var btn = new Button();
btn.Clicked += (s, e) => Console.WriteLine("Button clicked!");
btn.Click(); // output: Button clicked!
Visualize it
Final interview answer
A delegate is a type-safe function pointer — a variable that holds a reference to a method matching a specific signature. An event is a delegate with publisher-subscriber semantics enforced by the compiler: only the declaring class can invoke it, but any outside code can subscribe with +=. Func and Action are pre-built generic delegate types that save you from declaring custom delegates for 99% of cases — Func returns a value, Action returns void. Under the hood, lambda expressions are compiled into delegate instances. In real code I use Func/Action for callbacks and LINQ, event for UI interactions and domain events, and custom delegates only when the signature is complex enough to benefit from a named type.
Q5

What is boxing and unboxing? Why does it hurt performance?

C# FundamentalsMediumOriginal detailed notebook
Why asked

Shows you understand the CLR type system and can reason about performance. Common in LINQ-heavy and generic code reviews.

How it works

Boxing = wrapping a value type in an object reference so it can be stored on the heap. Unboxing = extracting the value back. Both copy memory and allocate. In a tight loop of a million integers, boxing creates a million heap objects.

Practical / Real-world context

ArrayList (pre-generics) boxed every int. Non-generic collections like Hashtable box value types. Calling object.Equals() on a struct boxes it. Modern code with List<T> avoids this, but it still lurks in logging frameworks and reflection.

Code example
C# / SQL / Angular
// Boxing — value type → object (heap allocation)
int x = 42;
object boxed = x;  // copies 42 onto heap, wraps in object

// Unboxing — object → value type (must cast + copy)
int y = (int)boxed;  // throws InvalidCastException if wrong type

// Hidden boxing in ArrayList (avoid!)
ArrayList old = new ArrayList();
old.Add(42);           // boxes int → object every time
int val = (int)old[0]; // unboxes every time

// Modern way: generics eliminate boxing
List<int> modern = new List<int>();
modern.Add(42);        // no boxing — List<int> is specialised
int v2 = modern[0];    // no unboxing

// Dangerous: loop boxing
for (int i = 0; i < 1_000_000; i++)
    object o = i;  // 1 million heap allocations!
Visualize it
Final interview answer
Boxing is the CLR operation of wrapping a value type (int, struct) in an object reference so it can be stored on the heap. Unboxing extracts the value back, requiring a type cast. Both operations involve heap allocation and memory copying, which the garbage collector then has to clean up. In hot paths — like sorting a million integers or filling an old ArrayList — boxing creates enormous GC pressure. Modern code avoids this by using generic collections like List&lt;T&gt; and Dictionary&lt;TKey, TValue&gt;, where the compiler generates type-specialised code with no boxing. I watch for hidden boxing in struct implementations of non-generic interfaces and in string.Format calls with value types.
Q6

What is async/await in C#? How does it work under the hood?

Async & MemoryHardOriginal detailed notebook
Why asked

Every ASP.NET Core API, every EF Core query, every HTTP call uses async/await. Microsoft interviewers WILL go deep on this.

How it works

async/await is syntactic sugar over a state machine. The compiler transforms an async method into a struct implementing IAsyncStateMachine. Each await point is a state transition. If the awaited task is not yet complete, the state machine suspends without blocking the thread — control returns to the caller. When the task completes, the continuation runs (usually back on the original synchronisation context in UI apps, or any thread pool thread in ASP.NET Core).

Practical / Real-world context

Database queries (EF Core: await db.Users.ToListAsync()), HTTP calls (await httpClient.GetAsync(...)), file I/O — all I/O-bound operations. While the DB is thinking, the thread is free to serve other requests.

Code example
C# / SQL / Angular
// Simple async method
public async Task<string> GetUserAsync(int id) {
    // awaiting = give up thread until DB responds
    var user = await _db.Users.FindAsync(id);
    return user?.Name ?? "Not found";
}

// What the compiler generates (simplified):
// A state machine struct with:
// State 0: start the DB task, if not complete → return
// State 1: resume when task completes, extract result

// ConfigureAwait(false) — don't capture sync context
// Use in library code to avoid deadlocks
public async Task<string> LibraryMethod() {
    var data = await File.ReadAllTextAsync("file.txt")
                         .ConfigureAwait(false);
    return data;
}

// Parallel async with Task.WhenAll
var [users, orders] = await Task.WhenAll(
    GetUsersAsync(),
    GetOrdersAsync()
);
Visualize it
Final interview answer
async/await is compiler syntactic sugar. The compiler transforms the async method into a state machine struct. Each await point becomes a state — if the awaited Task isn't complete yet, the method returns immediately, freeing the thread to do other work. When the Task completes, the runtime schedules the continuation to run from where it left off. Crucially, no thread is blocked while waiting for I/O — the thread is returned to the thread pool and can serve other requests. In ASP.NET Core this is how you handle thousands of concurrent DB calls with a small thread pool. I use ConfigureAwait(false) in library code to avoid deadlocks from sync-context capture, and Task.WhenAll when I can fire multiple async operations in parallel.
Q7

What is the difference between Task and ValueTask?

Async & MemoryHardOriginal detailed notebook
Why asked

Advanced question that shows you understand CLR allocation behaviour. Teams building high-throughput APIs care about this.

How it works

Task is a class — it always allocates on the heap even for completed results. ValueTask is a struct — if the result is already available (cache hit, synchronous path), it returns without any heap allocation. If it needs to go async, it wraps a Task internally.

Practical / Real-world context

IMemoryCache lookups — 90% of calls hit the cache and return synchronously. Using Task would allocate a heap object for every hit. ValueTask saves millions of allocations per minute in high-traffic APIs.

Code example
C# / SQL / Angular
// Task — always heap-allocated
public async Task<int> GetFromDbAsync() {
    await Task.Delay(100); // simulates DB
    return 42;
}

// ValueTask — zero alloc when synchronous
private int _cached = 0;
public ValueTask<int> GetCachedAsync() {
    if (_cached != 0)
        return ValueTask.FromResult(_cached); // NO heap allocation
    return new ValueTask<int>(FetchFromDbAsync()); // wraps Task
}

private async Task<int> FetchFromDbAsync() {
    _cached = await LoadFromDatabaseAsync();
    return _cached;
}

// Rules of ValueTask:
// 1. Await it only ONCE
// 2. Don't store it and await later
// 3. Use .AsTask() to convert if you need to share it
Visualize it
Final interview answer
Task is a class, which means every async method returning Task allocates a heap object even if the result is immediately available. ValueTask is a struct that can hold the result inline without heap allocation when the value is synchronous. This matters in high-throughput code where 80% of calls hit a cache — using Task would create millions of short-lived heap objects, stressing the garbage collector. ValueTask avoids that. The constraint is that you can only await a ValueTask once and you must not store it for later use. In ASP.NET Core and library code like IMemoryCache, ValueTask is the standard return type for this reason.

What is Dependency Injection (DI) in .NET Core and how does it work?

.NET CoreEasyOriginal detailed notebook
Why asked

DI is the backbone of every ASP.NET Core application. You will use it every day. Interviewers will probe your understanding of lifetimes.

How it works

The DI container (IServiceProvider) is a registry that maps service interfaces to concrete implementations. When you request a service (via constructor, controller action, or IServiceScopeFactory), the container creates and injects the correct implementation. Lifetime controls how long instances live.

Practical / Real-world context

ILogger injected into every controller, IRepository injected into services, DbContext scoped per HTTP request — all managed by DI. Without DI, you would manually new() everything and testing would require real databases.

Code example
C# / SQL / Angular
// 1. Register services in Program.cs
builder.Services.AddSingleton<ICache, RedisCache>();
builder.Services.AddScoped<IUserRepository, UserRepository>();
builder.Services.AddTransient<IEmailService, SmtpEmailService>();

// 2. Inject via constructor (preferred)
public class UserService {
    private readonly IUserRepository _repo;
    private readonly IEmailService _email;

    public UserService(IUserRepository repo, IEmailService email) {
        _repo = repo;   // container injects automatically
        _email = email;
    }
}

// 3. Lifetimes explained:
// Singleton  → one instance for app lifetime (config, cache)
// Scoped     → one instance per HTTP request (DbContext)
// Transient  → new instance every time (lightweight services)

// Captive dependency antipattern — AVOID:
// Singleton cannot depend on Scoped (Scoped has shorter life)
Visualize it
Final interview answer
Dependency Injection in .NET Core is a built-in IoC container. You register service interfaces with their implementations and a lifetime in Program.cs using AddSingleton, AddScoped, or AddTransient. The framework then constructs objects and injects dependencies automatically through constructors. Singleton creates one instance for the application lifetime — suitable for stateless services like caching. Scoped creates one instance per HTTP request — ideal for DbContext so the same unit-of-work is shared across a request. Transient creates a new instance every time it is requested — suitable for lightweight, stateless operations like email sending. The critical mistake to avoid is the captive dependency: a singleton depending on a scoped service, which means the scoped service is created once and never refreshed — causing data corruption or stale state bugs.

What is middleware in ASP.NET Core? How does the pipeline work?

.NET CoreMediumOriginal detailed notebook
Why asked

Every ASP.NET Core request passes through middleware. You need to understand the pipeline to debug issues, add auth, logging, CORS, or custom logic.

How it works

Middleware is a chain of components, each getting the HttpContext and a next() delegate. Each component can run code before AND after calling next(). The pipeline is configured in Program.cs with app.Use(), app.UseMiddleware<T>(), or app.Run(). Order matters — authentication must run before authorisation.

Practical / Real-world context

Logging every request/response, CORS headers, JWT auth, rate limiting, exception handling, response caching — all middleware. A custom middleware can short-circuit the pipeline and return early (e.g. reject unauthenticated requests).

Code example
C# / SQL / Angular
// Custom middleware class
public class RequestLoggingMiddleware {
    private readonly RequestDelegate _next;
    private readonly ILogger<RequestLoggingMiddleware> _logger;

    public RequestLoggingMiddleware(RequestDelegate next, ILogger<...> logger) {
        _next = next; _logger = logger;
    }

    public async Task InvokeAsync(HttpContext context) {
        _logger.LogInformation("→ {Method} {Path}", 
            context.Request.Method, context.Request.Path);
        
        await _next(context);  // call next middleware
        
        _logger.LogInformation("← {StatusCode}", 
            context.Response.StatusCode);
    }
}

// Register in Program.cs — ORDER MATTERS!
app.UseExceptionHandler();   // 1st — catches all errors
app.UseHttpsRedirection();   // 2nd
app.UseAuthentication();     // 3rd — who are you?
app.UseAuthorization();      // 4th — are you allowed?
app.UseMiddleware<RequestLoggingMiddleware>();
app.MapControllers();        // last — route to action
Visualize it
Final interview answer
In ASP.NET Core, every HTTP request flows through a pipeline of middleware components. Each middleware receives the HttpContext and a next delegate. It can run code before calling next (request processing) and after (response processing). The order of registration in Program.cs is critical — UseExceptionHandler must be first to catch anything, UseAuthentication before UseAuthorization, and routing last. Any middleware can short-circuit the pipeline by not calling next, which is how authentication rejects unauthorised requests. I use custom middleware for cross-cutting concerns like request logging, correlation IDs, and tenant resolution that need to run on every request without polluting controller code.

What is the difference between .NET Framework, .NET Core, and .NET 5/6/7/8?

.NET CoreEasyOriginal detailed notebook
Why asked

Orientation question. Shows you understand the platform history and can advise teams on migration.

How it works

.NET Framework (2002–present): Windows-only, ships with Windows, not open source. .NET Core (2016–2020): cross-platform, open-source, modular. .NET 5+ (2020–present): unified platform — .NET Core + Xamarin + Mono under one roof, major version each November. No more "Core" branding.

Practical / Real-world context

If you join a company with legacy enterprise apps on .NET Framework 4.8, you will help migrate them. New projects should target .NET 8 (LTS). Understanding the differences saves days of debugging compatibility issues.

Code example
C# / SQL / Angular
// .NET Framework — Windows GAC, full System.Web
using System.Web.Mvc; // only on .NET Framework

// .NET Core / .NET 5+ — cross-platform
using Microsoft.AspNetCore.Mvc; // works everywhere

// Key differences in project files:
// .NET Framework:
<TargetFramework>net48</TargetFramework>

// .NET 8:
<TargetFramework>net8.0</TargetFramework>

// Check runtime version at runtime
Console.WriteLine(RuntimeInformation.FrameworkDescription);
// .NET 8.0.x  or  .NET Framework 4.8
Visualize it
Final interview answer
.NET Framework is the original Windows-only runtime that ships with Windows and has full backward compatibility but cannot run on Linux or Mac. .NET Core (2016–2020) was the cross-platform, open-source rebuild. Starting with .NET 5 in 2020, Microsoft unified everything under a single ".NET" brand — dropping the "Core" — with a major release every November. Even-numbered versions are LTS (three years support); odd-numbered are STS (eighteen months). For new projects I always target the latest LTS — currently .NET 8. Legacy .NET Framework code can still run but does not get new features.
Q11

What is the Entity Framework Core and how does the DbContext work?

.NET CoreMediumOriginal detailed notebook
Why asked

95% of .NET backend apps use EF Core for data access. You MUST understand change tracking, migrations, and query execution.

How it works

EF Core is an ORM (Object-Relational Mapper). DbContext is your unit-of-work and session with the database. It tracks changes to your entities (Added, Modified, Deleted, Unchanged). When you call SaveChanges(), it generates SQL for all tracked changes and executes them in one transaction.

Practical / Real-world context

UserService creates a new User entity, sets properties, calls dbContext.Users.Add(user), then dbContext.SaveChangesAsync() — EF generates an INSERT. Change tracking means you don't need to call Update() if you've already queried the entity.

Code example
C# / SQL / Angular
// DbContext setup
public class AppDbContext : DbContext {
    public DbSet<User> Users { get; set; }
    public DbSet<Order> Orders { get; set; }

    protected override void OnModelCreating(ModelBuilder mb) {
        mb.Entity<User>().HasMany(u => u.Orders)
          .WithOne(o => o.User).HasForeignKey(o => o.UserId);
    }
}

// Create — EF generates INSERT
var user = new User { Name = "Alice", Email = "a@b.com" };
db.Users.Add(user);
await db.SaveChangesAsync(); // INSERT INTO Users...

// Read — generates SELECT
var users = await db.Users
    .Where(u => u.IsActive)
    .Include(u => u.Orders)     // JOIN
    .AsNoTracking()             // read-only, no change tracking
    .ToListAsync();

// Update — change tracking
var user = await db.Users.FindAsync(1);
user.Name = "Bob";          // EF detects change
await db.SaveChangesAsync(); // UPDATE Users SET Name='Bob'

// N+1 query problem — AVOID:
foreach (var user in users) {
    var count = user.Orders.Count(); // hits DB per user!
}
// FIX: use Include() or load in one query
Visualize it
Final interview answer
Entity Framework Core is an ORM that maps C# objects to database tables. The DbContext is both your session with the database and your unit-of-work — it tracks every entity you have queried or added in memory. It uses a change tracker to detect modifications: if you query a user and change their name, EF knows to generate an UPDATE when you call SaveChangesAsync. For read-only queries I always use AsNoTracking, which skips the change tracker and uses less memory. I use Include for eager loading to avoid the N+1 query problem, where each entity in a loop triggers a separate database round trip. Migrations are version-controlled incremental schema changes generated by EF Core from model changes.
Q12

Explain the SOLID principles with real examples.

OOP & DesignMediumOriginal detailed notebook
Why asked

SOLID is the design philosophy behind clean .NET codebases. Microsoft-level interviews always probe this — not just the acronym, but whether you apply it.

How it works

S: Single Responsibility — one class, one reason to change. O: Open/Closed — open for extension, closed for modification. L: Liskov Substitution — subtypes must be usable where base types are expected. I: Interface Segregation — small focused interfaces. D: Dependency Inversion — depend on abstractions, not concretes.

Practical / Real-world context

S: UserService should not also send emails. O: Add a new payment method by adding a new class, not modifying existing PaymentProcessor. L: If Square extends Rectangle but breaks the area formula, it violates LSP. I: IWorker vs splitting into IDoable + IFeedable. D: Controller depends on IUserRepository, not UserRepository.

Code example
C# / SQL / Angular
// S — Single Responsibility
// BAD:
class UserService {
    public void CreateUser(User u) { }
    public void SendWelcomeEmail(User u) { } // wrong responsibility!
}
// GOOD: split into UserService + EmailService

// O — Open/Closed
interface IDiscount { decimal Apply(decimal price); }
class PercentDiscount : IDiscount { ... }
class FlatDiscount    : IDiscount { ... }
// Add new discount type by adding a class, not editing existing ones

// L — Liskov Substitution
class Bird { public virtual void Fly() { } }
class Penguin : Bird {
    public override void Fly() => throw new Exception(); // VIOLATES LSP!
}
// Fix: separate IFlyable interface

// I — Interface Segregation
// BAD: one fat interface
interface IWorker { void Work(); void Eat(); void Sleep(); }
// GOOD: small focused interfaces
interface IWorkable { void Work(); }
interface IFeedable { void Eat(); }

// D — Dependency Inversion
// BAD: depends on concrete class
class OrderService { private SqlOrderRepo _repo = new(); }
// GOOD: depends on abstraction, injected via DI

             where u.IsActive
             orderby u.Name

What is the Garbage Collector in .NET and what are the generations?

// _stream disposed here automatically

The .NET garbage collector is a generational, compacting collector. Gen 0 holds newly allocated short-lived objects and is collected very frequently and cheaply. Final interview answer
